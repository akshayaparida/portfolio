## Code of Conduct

This project and everyone participating in it is governed by our Code of Conduct. By participating, you are expected to uphold this code. Please report unacceptable behavior to [email].

## How Can I Contribute?

### Reporting Bugs

Bugs are tracked as [GitHub issues](https://github.com/akshayaparida/portfolio/issues). When reporting a bug, please follow the bug report template and provide as much detail as possible.

Before creating a bug report, please check that an issue doesn't already exist for the same problem. If it does, feel free to add your experience to the existing issue.

### Suggesting Enhancements

Enhancement suggestions are also tracked as [GitHub issues](https://github.com/akshayaparida/portfolio/issues). When suggesting an enhancement, please follow the feature request template and provide detailed information about your proposal.

Before creating an enhancement suggestion, please check that an issue doesn't already exist for the same idea. If it does, feel free to add your thoughts to the existing issue.

### Your First Code Contribution

Unsure where to begin contributing? You can start by looking through these `beginner` and `help wanted` issues:

- [Beginner issues](https://github.com/akshayaparida/portfolio/issues?q=is%3Aopen+is%3Aissue+label%3Abeginner) - issues which should only require a few lines of code
- [Help wanted issues](https://github.com/akshayaparida/portfolio/issues?q=is%3Aopen+is%3Aissue+label%3Ahelp+wanted) - issues which should be a bit more involved than `beginner` issues

## Development Setup

1. Fork the repository
2. Clone your fork
3. Create a new branch for your changes
4. Install dependencies: `pnpm install`
5. Start the development server: `pnpm dev`

## Pull Requests

1. Follow the [style guides](#style-guides) outlined below
2. Update the README.md with details of changes if applicable
3. Add tests for any new features
4. Ensure all tests pass
5. Submit your pull request

## Style Guides

### Git Commit Messages

- Use the present tense ("Add feature" not "Added feature")
- Use the imperative mood ("Move cursor to..." not "Moves cursor to...")
- Limit the first line to 72 characters or less
- Reference issues and pull requests liberally after the first line

### JavaScript/TypeScript Style Guide

- Follow the ESLint configuration provided in the project
- Use meaningful variable and function names
- Write comments for complex logic
- Keep functions small and focused

### CSS/Tailwind Style Guide

- Use Tailwind utility classes where possible
- Follow the design system defined in the Tailwind configuration
- Keep custom CSS to a minimum
- Use semantic class names when creating components

### Documentation Style Guide

- Use clear, concise language
- Follow the templates provided for issues and pull requests
- Include examples where helpful
- Keep documentation up-to-date with code changes

## Issue Labels

We use the following labels to categorize issues:

- `bug` - Something isn't working
- `documentation` - Improvements or additions to documentation
- `duplicate` - This issue or pull request already exists
- `enhancement` - New feature or request
- `good first issue` - Good for newcomers
- `help wanted` - Extra attention is needed
- `invalid` - This doesn't seem right
- `question` - Further information is requested
- `wontfix` - This will not be worked on

## Questions?

If you have any questions about contributing, feel free to open an issue with the `question` label or reach out to the maintainers.
